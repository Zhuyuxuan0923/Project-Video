================================================================================
                    AI短剧自动生成系统 - 项目总结
================================================================================

1. 项目概述
--------------------------------------------------------------------------------
本项目构建了一个基于 Celery 分布式任务队列的 AI 短剧自动生成系统。用户只需在终端
输入一个主题，系统便会自动完成「文案→图片→视频」的全流程生成，中途任何一步失败
都会自动重试，可通过 Flower Web 面板实时监控整个运行状态。


2. 技术架构
--------------------------------------------------------------------------------

  组件              技术选型               职责
  ─────────────────────────────────────────────────
  Master 调度器      Python + Celery        接收用户请求，按序编排三个 Worker
  文案 Worker        Celery Task            调用 AI 生成短剧剧本
  图片 Worker        Celery Task            为每个场景生成配图
  视频 Worker        Celery Task            将图片+音频合成为视频
  消息中间件         Redis 7                可靠的消息传递 (Broker + Backend)
  监控面板           Flower 2.0             Web 端实时查看任务状态
  AI 后端            可插拔架构 (Mock默认)   支持接入真实 OpenAI 等 API
  API Scout          实时 Web 检索         自动搜索当前最新最优 API 并推荐


3. 工作流
--------------------------------------------------------------------------------

  用户在终端输入主题
         │
         ▼
  ┌─────────────────┐
  │  cli.py         │  Click CLI 解析参数
  └────────┬────────┘
           ▼
  ┌─────────────────┐
  │  orchestrator   │  Master 调度器：逐阶段编排
  └────────┬────────┘
           │ submit via Redis
           ▼
  ┌─────────────┐    ┌──────────────┐    ┌──────────────┐
  │  copywriting│───▶│  image_gen   │───▶│  video_gen   │
  │  Worker     │    │  Worker      │    │  Worker      │
  └─────────────┘    └──────────────┘    └──────────────┘
       │                    │                    │
       └────────────────────┴────────────────────┘
                            │
                            ▼
                   ┌──────────────┐
                   │  Flower :5555│  实时可视化监控
                   └──────────────┘


4. 核心特性
--------------------------------------------------------------------------------

  [分布式] 三个专用 Worker，各自监听独立队列 (copywriting/image_gen/video_gen)
  [编排型] Master 调度器按序驱动：前一步成功才进入下一步，阶段间数据自动传递
  [容错性] 每个任务 max_retries=3，指数退避 60s，幂等设计支持安全重试
  [可观测] Flower 面板实时展示任务进度、成功/失败/重试状态、Worker 心跳
  [可扩展] AI 后端采用抽象基类 + 注册机制，切换到真实 API 只需新增一个类
  [智能选型] API Scout 实时检索当前最新 API 市场，自动推荐最优模型


5. 项目文件清单
--------------------------------------------------------------------------------

  config.py              Celery 全局配置 (broker、backend、队列、重试策略)
  celery_app.py          Celery 应用实例 (队列路由、任务注册)
  orchestrator.py        Master 调度器 (阶段编排、状态轮询、结果展示)
  cli.py                 CLI 命令行入口 (run / status / worker / scout)
  api_scout.py           API 实时检索推荐 (DuckDuckGo 免费搜索，无需 Key)
  docker-compose.yml     Docker 编排 (Redis + Flower)
  requirements.txt       Python 依赖
  .env.example           环境变量模板

  tasks/
    __init__.py          任务模块导出
    copywriting.py       文案生成任务 (3次重试, 60s退避)
    image_gen.py         图片生成任务 (3次重试, 60s退避)
    video_gen.py         视频合成任务 (3次重试, 60s退避)

  ai_backends/
    __init__.py          后端注册表 (get_backend 工厂函数)
    base.py              AI 后端抽象基类 (generate_script / generate_image / generate_video)
    mock.py              Mock 后端实现 (快速验证流程)


6. 开发环境
--------------------------------------------------------------------------------

  Python:    3.14.4
  Celery:    5.6.3
  Redis:     7.x (Docker)
  Flower:    2.0.1
  Platform:  Windows 11 + Docker Desktop
  Shell:     Git Bash


7. 已知问题与注意事项
--------------------------------------------------------------------------------

  [1] Celery 5.6.x + Python 3.14 + Windows 的 prefork 池存在 trace_info 兼容
      性问题，当前使用 solo 池作为 workaround。生产环境建议 Linux + prefork。

  [2] Windows GBK 终端会导致 emoji 和部分 Unicode 字符显示异常，当前代码
      已改为 ASCII 安全字符。

  [3] 多个 Worker 进程同时运行时会复用电 hostname，产生 DuplicateNodename
      警告。已通过 solo 池规避，后续可配置唯一 hostname 彻底解决。

  [4] Mock AI 后端返回固定示例数据，生产使用需替换为真实 API 实现。
