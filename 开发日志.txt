Phase 1: 架构设计与评审
================================================================================

  1.1 需求分析
      - 梳理四大核心服务：Master调度器、三个专用Worker、Redis中间件、Flower监控
      - 确定数据流方向：用户输入主题 → 文案 → 图片 → 视频 → 输出结果
      - 明确容错需求：每步失败自动重试，整个链路状态可观测

  1.2 技术选型
      - Celery 5.x: Python 生态最成熟的分布式任务队列
      - Redis 7: Celery 官方推荐的 Broker + Result Backend
      - Flower 2.0: Celery 配套监控工具，开箱即用
      - Click + Rich: 终端 CLI 框架
      - 可插拔 AI 后端: 抽象基类 + Mock 默认实现

  1.3 项目结构设计
      - config.py: 集中管理所有 Celery 配置
      - celery_app.py: 单例 Celery 应用实例
      - tasks/: 三个独立 Worker 任务模块
      - ai_backends/: 可插拔 AI 后端 (base + mock + 未来扩展)
      - orchestrator.py: Master 调度器
      - cli.py: CLI 入口

  1.4 关键设计决策
      - 采用三个独立队列而非单队列：支持未来不同 Worker 独立扩缩容
      - 手动编排代替 Celery Chain：更稳健、更可调试、每步状态可见
      - solo 池代替 prefork：规避 Python 3.14 + Windows 兼容性问题

Phase 4: Worker 任务实现
================================================================================

  4.1 文案生成 (tasks/copywriting.py)
      - @shared_task(bind=True, max_retries=3, default_retry_delay=60)
      - 输入: topic (str)
      - 调用: backend.generate_script(topic)
      - 异常处理: 捕获所有异常，自动重试，超限后抛出 MaxRetriesExceededError

  4.2 图片生成 (tasks/image_gen.py)
      - 输入: script_result (dict) - 上一步的完整输出
      - 遍历 scenes 列表，为每个场景调用 backend.generate_image()
      - 在输入 dict 上追加 images 字段后返回

  4.3 视频合成 (tasks/video_gen.py)
      - 输入: script_result (dict) - 包含 scripts + images
      - 调用 backend.generate_video(scenes, images)
      - 组装最终输出结构: {topic, title, scenes, images, narration, video}

Phase 7: 端到端调试验证
================================================================================

  7.1 环境准备
      - 安装 Python 依赖 (celery, redis, flower, click)
      - 启动 Docker Desktop
      - docker-compose up -d redis (启动 Redis 7 容器)

  7.2 问题排查与修复

      问题1: GBK 编码异常
      ├─ 现象: emoji 字符 (⭕▶️✅❌♻️⭐) 无法在 Windows GBK 终端输出
      └─ 修复: 全部替换为 ASCII 安全字符 ([...] [>] [OK] [XX] [~] [*])

      问题2: Celery chain 在 Python 3.14 上回调失败
      ├─ 现象: fast_trace_task 中 _localized 为空列表，解包失败
      ├─ 根因: Celery 5.6.3 与 Python 3.14 的 trace 优化存在兼容性 bug
      └─ 修复: 放弃 Celery chain，改为 orchestrator 手动分发 + 轮询编排

      问题3: Celery prefork 池在 Windows 上子进程失败
      ├─ 现象: billiard pool 子进程中 _loc 为 None
      └─ 修复: 改用 -P solo 单进程池 (对于 demo 场景性能够用)

      问题4: 多个僵尸 Worker 进程导致任务路由混乱
      ├─ 现象: 任务有时成功有时失败，行为不稳定
      ├─ 根因: 之前调试时启动的 Worker (prefork+threads+solo) 全部留存
      │         在 Redis 中产生冲突的节点注册和队列消费
      └─ 修复: 批量 kill 僵尸进程，统一使用 solo 池重启

  7.3 最终验证
      - 测试主题: "赛博朋克武侠"
      - 文案生成: 2s ✓
      - 图片生成: 5s (3张图) ✓

经验总结
================================================================================

  1. Celery 在新 Python 版本上可能存在兼容性风险，建议生产环境使用成熟的
     Python+Celery 版本组合 (如 Python 3.11 + Celery 5.3)。

  2. 手动编排比链式调用更稳健。虽然 Celery chain/chord 语法优雅，但在
     异常处理和跨队列场景下可控性差，项目最终采用手动编排是正确选择。

  3. 进程管理很重要。多次调试启动的 Worker 如不销毁会互相干扰，导致诡异的
     间歇性故障。应在每次测试前检查并清理残留进程。

  4. 可插拔架构代价小收益大。AI 后端抽象只花了约 30 行代码，但为后续接入
     真实 API 铺平了道路，无需修改任何 Worker 或调度器代码。

  5. ASCII-safe 文案是跨平台 CLI 工具的最佳实践，避免依赖 emoji 或特殊
     Unicode 字符。

  6. API Scout 证明：在 AI 模型快速迭代的背景下，工具化的实时检索比写死
     推荐更可持续。130 行代码换来自动选型能力，性价比极高。
